# Decisions Log — google-reviews-download

Append-only ADR-style log. One line per decision unless rationale is non-obvious.

## 2026-04-28

- **D-001** Project = EMD targeting "google reviews download" + variants. Cloned the operational setup of `halflife` (planning bundle, scheduled-routine, wrapper-publishes model, corgi $1/wk cap, Obsidian mirror).
- **D-002** SemanticForce is the only data source. No direct scraping. Mock-first: when `SF_API_KEY` is unset, the client returns fixture data from `mocks/semanticforce/*.json`. Real creds arrive via a human-gated Phase 4 leaf.
- **D-003** SF client surface area is **our** typed contract (`lib/semanticforce/types.ts`), not SF's. When the real schema lands, we adapt internally; callers and exporters never change.
- **D-004** Stack starting position: Next.js 15 App Router + TS + Tailwind + shadcn/ui on Vercel; Vercel KV for cache; SheetJS (`xlsx`) for XLSX export. Re-evaluatable in Phase 1.
- **D-005** CSV defaults: UTF-8 with BOM, CRLF, QUOTE_ALL (per project-memory `feedback_csv_ascii_for_excel`). XLSX gets unicode natively.
- **D-006** Daily autonomous routine, ≤10 commands/run, 04:00 Europe/Madrid (one hour after `halflife-nightly`), wrapper-publishes (no `git push` / `gh pr create` from the agent). Routine forbidden from running installs, deploying, buying domains, or sending external messages.
- **D-007** Cache strategy: Vercel KV keyed by normalised `place_id`, 24h TTL. Reviews are slow-changing; 24h amortises cost without staleness pain.

## 2026-05-09

- **D-008** Tailwind **v3.4** (not v4) for the manifest baseline. Rationale: shadcn/ui's component generators and most third-party Tailwind ecosystem snippets still assume v3 syntax; we can revisit v4 once shadcn's v4 path is the default. `tailwindcss-animate` is included because shadcn components reference it.
- **D-009** `xlsx` pinned to `^0.18.5` from the npm registry rather than the SheetJS CDN tarball (`https://cdn.sheetjs.com/...`). Rationale: keeps `package.json` registry-only and lockfile-friendly for now; the npm 0.18.5 build is sufficient for our writer (one row per review, frozen header, column widths). If a future SheetJS feature is needed we'll switch to the CDN tarball at that point — the import surface (`import * as XLSX from 'xlsx'`) is identical either way.
- **D-010** React 19 + Next.js 15 (App Router) confirmed in the manifest. Node engine `>=20.11` to match Next 15's minimum.
- **D-011** `tsconfig.json` uses `moduleResolution: "bundler"` (not `node`) and `module: "esnext"`. Rationale: this is the Next 15 default and is required for React Server Components / `next/font` / `next/image` type resolution to work cleanly. `paths: { "@/*": ["./*"] }` matches the shadcn/ui import alias convention.
- **D-012** `next.config.ts` (TS, not JS) is intentionally minimal — only `reactStrictMode: true`. We add config knobs as concrete needs surface (image domains, redirects, headers) rather than pre-configuring speculatively.
- **D-013** `.env.example` documents `SF_API_KEY` (blank by default → fixture mode), `SF_API_BASE` (defaulted to `https://api.semanticforce.net`, refined when real creds land), the four `KV_*` vars Vercel KV provisions automatically, and `NEXT_PUBLIC_PLAUSIBLE_DOMAIN`. `.env.local` is gitignored; `.env.example` is the only env file in the repo.
- **D-014** Mid (80) and large (500) fixtures are produced by a deterministic, seeded generator (`scripts/gen-fixtures.py`) rather than hand-written. Rationale: 500 hand-authored reviews are unmaintainable, and a seeded `random.Random` makes the JSON byte-stable across reruns so diffs only appear when we deliberately change the model. The script lives outside the Next.js source tree so it never ships to Vercel; review distribution targets — 13 languages with ~40% en, rating skew 60/20/10/5/5 across 5★→1★, ~15% photos, owner-response chance increasing with negativity (5★ 3% → 1★ 70%) — are encoded in the script and can be tuned in one place. `small-business.json` stays hand-curated so the small fixture remains the easiest one to read end-to-end during debugging.
- **D-015** SF client cursor is `btoa(JSON.stringify({offset:N}))` for the fixture path, and an opaque pass-through for the real API path. Rationale: keeps the fixture client edge-runtime safe (no `Buffer`), and the offset-cursor is only meaningful in fixture mode — the real SF cursor is whatever SF returns and we never inspect it. We also clamp `limit` to `[1, 100]` (default 50) inside the client so callers can't accidentally explode the request.
- **D-016** SF errors surface as a single `SemanticForceError` class with a coarse `code` enum (`rate_limited` | `not_found` | `unauthorized` | `bad_request` | `upstream_error` | `unknown`) plus the upstream HTTP status when available. Rationale: route handlers and exporters need to map "what went wrong" to user-facing messaging without parsing free-form strings; the enum is intentionally coarse so it survives whatever SF's real error envelope ends up looking like in L4.1. Network failures map to `upstream_error`, mirroring 5xx, so retry policy is one branch.

## 2026-05-10

- **D-017** `place_id` normalisation produces a `{raw, slug}` pair: `raw` is the canonical case-sensitive ID we send upstream, `slug` is the lowercased, URL-safe key used in cache keys, share URLs, and filenames. Rationale: Google Place IDs are case-sensitive after the prefix, but we want stable cache keys regardless of how the user pasted the URL. Two keys (raw + slug) is cheaper than re-deriving slug at every cache touch, and avoids the bug where a mixed-case paste misses a cache that a lowercase paste populated.
- **D-018** Short-link resolution (`maps.app.goo.gl/...`) is **out** for v1. Resolving costs one HEAD/GET per submit, which we are not paying for at MVP and which would add a failure mode (Google rate-limits the redirect endpoint). UI surfaces "paste the long URL or the Place ID" instead.
- **D-019** Pagination walker uses `limit: 100` per page (the max), `HARD_CAP_REVIEWS = 5_000`, `HARD_CAP_PAGES = 50`. Reviews assembled into a single payload before caching; per-page slices are not cached. Rationale: SF is the rate-limit dimension, not us; sequential walking gives correct ordering and respects upstream backpressure; a single assembled payload means one KV read per cache hit.
- **D-020** Cache key shape: `gr:reviews:v1:<slug>`. The `v1:` segment is a schema version we bump (to `v2:`) when the cached payload shape changes — old keys then expire on TTL rather than serve stale-shape data into a new code path. Cheaper than a flag-day migration. Errors and partial walks are **not** cached so the next request retries against SF.
- **D-021** Rate-limiter is in-memory token bucket (10/min/IP, burst 10) at the edge, not KV-backed. Rationale: KV-backed adds one RTT per request for a problem we do not have at MVP; per-region in-memory state is sufficient until traffic warrants a swap. Swap is a one-leaf change at the same interface if needed (Sprint 2). 429 envelope mirrors `SemanticForceError` shape so UI handles upstream-rate-limited and our-rate-limited identically.
- **D-022** L1.6 split into L1.6a (agent — candidate variants only, no volumes) and L1.6b (human-gated — local `corgi keywords` pass to add volumes/competition). Rationale: the routine cannot call DataForSEO (no creds, would also blow the LEDGER cap if it could), so an "agent commits the candidate list, human runs corgi" partial-deferral was already the intent; splitting makes that explicit in the roadmap and lets L1.6a auto-merge tonight without leaving the leaf in a half-checked state. L3.1's `(depends on L1.6)` pointer is updated to `L1.6b` so the dependency is unambiguous.
- **D-023** SEO variants picked by three filters (buyer intent, tool-shaped SERP gap, fulfillable as-is) and grouped by intent (format-named / verb-led / surface-named). Rationale: grouping makes corgi-side comparisons apples-to-apples, and the filters keep us off variants that imply features we deliberately don't ship in v1 (sentiment, monitoring, multi-place batch — those are parked in `docs/seo-variants.md` "Out-of-scope variants" so future scope reviews don't re-discover them from scratch). Final pick of 5 happens in L3.1 after corgi data lands; agent does not pre-rank.
- **D-024** ESLint stays on the legacy `.eslintrc.json` config format rather than ESLint 9's flat config. Rationale: `eslint-config-next` 15.1 still ships its presets in the legacy shape, and using flat config would require adding `@eslint/eslintrc` + a `FlatCompat` shim just to load `next/core-web-vitals` and `next/typescript`. The legacy file is one line of `extends` and `next lint` accepts it natively — we revisit when Next.js's lint integration goes flat-only.
- **D-025** App skeleton (L2.1) ships the bare minimum: `app/layout.tsx`, `app/page.tsx` stub, `app/globals.css` with shadcn HSL CSS variables (zinc base, light + dark), Tailwind v3 + PostCSS configs, `lib/utils.ts` (`cn`), and `components.json` for shadcn generators. No fonts (Tailwind's default sans stack is fine until a designer asks for Geist/Inter); no shadcn components copied yet — they land per L2.4/L2.5 when the form and result preview need them. Skeleton goal is "next dev runs and shows the placeholder page once a human installs", nothing more.
- **D-026** `app/api/reviews/route.ts` runs on the **edge runtime** (`export const runtime = "edge"`). Rationale: the fixture client and `fetch`-based real client are both edge-safe (D-015 deliberately picked `btoa` over `Buffer` for exactly this), edge gives us per-region cold-start sub-100ms which matters for the "paste → preview" UX, and the future rate-limit middleware (L2.8) is already specified as edge — keeping the route on the same runtime avoids a node↔edge serialization seam. The JSON fixture imports in `client.ts` are bundled at build time so they don't pay a runtime cost.
- **D-027** Route response shapes are stabilised here so cache (L2.3), exporters (L2.6/L2.7), and UI (L2.4/L2.5) can all be built against a fixed envelope without re-derivation. Success body: `{ place: PlaceMeta, reviews: Review[], fetched_at: ISO, truncated?: true }` — `fetched_at` is the assembled-payload timestamp so cached responses can surface "stale by Nh" in the UI; `truncated: true` (only ever the literal `true`, never `false`) fires when either `HARD_CAP_REVIEWS` or `HARD_CAP_PAGES` from methodology §2 trips. Error body: `{ error: { code, message } }` — the same shape `SemanticForceError` produces, so UI dispatch is one branch (code values: SF's six, plus the route-layer additions `bad_request` for input parse failures and `not_implemented` for csv/xlsx until L2.6/L2.7 land). 429 mid-walk rate-limit body: error envelope + `partial: Review[]` (whatever we collected before SF rate-limited us) + optional `retry_after_s` (parsed from the SF error message, default 30s) and a matching `Retry-After` header.
- **D-028** Format dispatch in the route returns `501 not_implemented` for `csv` and `xlsx` until L2.6/L2.7 land their writers. Rationale: a 501 with an explicit pointer at the relevant leaf is more honest than a placeholder that silently emits JSON-with-wrong-MIME or a half-written CSV. The UI (L2.4/L2.5) will route to `format=json` for now and we'll flip the form's CSV/XLSX buttons live in the same PR that ships the writer.

## 2026-05-11

- **D-029** Reviews cache (L2.3) talks to Vercel KV via the **REST pipeline endpoint** (raw `fetch` POST with body `["GET", key]` / `["SET", key, value, "EX", ttl]`) rather than the `@vercel/kv` SDK. Rationale: zero new dependencies, edge-runtime safe by construction, identical wire calls to what the SDK does. When KV envs are unset (the routine's default fixture mode) the factory returns a process-local `Map` so the route still works end-to-end without any cache wiring — useful for local dev, fixture-only CI, and the autonomous routine. Cache write failures are swallowed (best-effort): a KV outage degrades to a cache miss, not a 500 on the user. Cache reads are wrapped in the same try/catch so a transient KV error never breaks the request — we just walk SF.
- **D-030** Cache slicing happens **after** the cache layer, not before. We persist the full assembled walk (pre-`userLimit`) so a `?limit=50` request and a later `?limit=200` request share one KV entry. Rationale: the user's `limit` is a display knob; the SF-cost-bearing dimension is the walk itself. Caching the post-limit slice would force a re-walk whenever a different limit comes in — defeats the cache. Matches methodology §2 "the per-request `limit` cap is final-response only". Truncation from HARD_CAP is also cacheable (it's a stable outcome of the walker); only mid-walk rate-limit partials are excluded.
- **D-031** Cache HIT/MISS surfaced as an `X-Cache: HIT|MISS` response header on success. Rationale: gives the UI (L2.5) and any future ops debugging a one-glance signal without polluting the JSON body shape. Not a load-bearing contract — only the body is part of the public API envelope (D-027); the header is for diagnostics.
- **D-032** L2.4 input form is a plain server-rendered `<form action="/api/reviews" method="GET" target="_blank">` — no `'use client'`, no JS state. Rationale: the smallest thing that satisfies the leaf (single field, format toggle, download button) is a native form whose submission the browser turns into `?placeId=…&format=…` and either renders (JSON) or downloads (CSV/XLSX once L2.6/L2.7 land) in a new tab. `target="_blank"` keeps the form mounted so a user can immediately tweak inputs and retry. L2.5's preview/result component will replace this with a client wrapper that fetches the JSON, renders the first 5 + total count, and offers an explicit download button against the same URL — that's the right point to introduce client-side state, not now.
- **D-033** CSV row schema (L2.6) is flat, denormalised, one row per review — but the place metadata (`place_name`, `place_id`, `place_url`) is repeated as the first three columns on every row. Rationale: a consultant who concats multiple exports for cross-place analysis needs a join key, and repeating three short strings is cheaper than telling them to lift it out of the filename. Photos collapse to `photo_count` + `photo_urls` (URLs joined with ` | `, single line); width/height are dropped because spreadsheets don't act on them. Owner response collapses to `owner_response_text` + `owner_response_at`. The `text` column preserves embedded newlines — CSV permits CR/LF inside a quoted field and Excel renders it as a wrapped cell, so QUOTE_ALL plus BOM+CRLF survives multi-paragraph reviews without an import wizard. Filename `google-reviews-<slug>-<YYYYMMDD>.csv` uses `payload.fetched_at`'s date (not wall clock) so a cache-hit download labels itself by the data vintage. The `limit` query param trims the file rows too — a `limit=50` request produces a 50-row file, even though the cache holds the full walk (D-030 unchanged).
- **D-035** Edge rate-limit middleware (L2.8) identifies clients by `x-forwarded-for` leftmost, falling back to `x-real-ip`, then `"unknown"` — **not** `request.ip` as methodology §4 originally wrote. Rationale: Next.js 15 removed `NextRequest.ip` (it was always platform-dependent guesswork); the two-header chain is what Vercel's edge actually sets. Bucket state lives in a module-level `Map<string, {tokens, updated_at}>` keyed by `${ip}:${pathname}` per §4. Stale entries (>10min idle) are evicted on next touch for the same key — passive eviction, no timer, edge-safe. `Retry-After` is the documented flat 6s (one token's worth at 10/min refill); a dynamic value based on bucket state was rejected as nice-to-have that diverges from the contract. Scope is locked via `config.matcher: ["/api/reviews"]` so the home page and SEO variants stay unmetered as §4 specifies. 429 body uses the same `{error: {code, message}}` envelope `SemanticForceError` produces (D-027) so the UI dispatches upstream and our-side rate-limits through one branch.
- **D-034** XLSX writer (L2.7) reuses `CSV_COLUMNS` from `lib/export/csv.ts` verbatim instead of declaring its own column list. Rationale: the two writers are intentional 1:1 surfaces (D-033 / ADR-003); a divergence would surprise a user who exports both formats for the same place. Sharing the constant pushes any future column edit through one place. `rating` is emitted as a JS `number` so XLSX gives it a numeric cell (sortable, summable in Excel); every other field is a string. Frozen header is set via **both** `ws['!views']` (canonical OOXML SheetView pane) and `ws['!freeze']` (SheetJS legacy convenience prop) — belt-and-braces because the published @types stop short of either, and we can't run an integration test without `npm install`; setting both costs nothing and survives whichever serializer path SheetJS picks. `XLSX.write({type: "array"})` returns an `ArrayBuffer`/`Uint8Array` — edge-safe, no Node `Buffer` escape — so the route on `runtime = "edge"` (D-026) hands the bytes straight to `NextResponse`. Content-Type pinned to the canonical OOXML MIME (`application/vnd.openxmlformats-officedocument.spreadsheetml.sheet`); filename mirrors `csvFilename` so CSV+XLSX downloads sort next to each other.

### ADR-001 — Stack: Next.js 15 + TS + Tailwind + shadcn/ui on Vercel, with `xlsx` for XLSX

**Status:** Accepted (2026-05-10). Supersedes nothing. Folds in D-004, D-008, D-009, D-010, D-011, D-012.

**Context.** The product is a single-page tool: paste a place, get a downloadable export. We need (a) a server endpoint that talks to SemanticForce and writes CSV/JSON/XLSX, (b) a small interactive UI, (c) ~10 SEO-rendered variant pages for long-tail queries, (d) an edge-friendly cache layer keyed by `place_id`. Hosting must be cheap-to-zero at MVP traffic and support edge runtime so the tool feels fast in EU and US. Two of us touch this codebase: Andrei and the autonomous routine. The routine writes one leaf per fire and must work without `npm install` between commits — so the stack has to be one where reading manifests + types is enough to keep iterating, even before a human installs packages.

**Decision.**
- **Framework:** Next.js 15 (App Router) on Vercel. React 19. Node `>=20.11`.
- **Language:** TypeScript with `moduleResolution: "bundler"`, `module: "esnext"`, `paths: { "@/*": ["./*"] }`.
- **Styling:** Tailwind CSS **v3.4** (not v4) with `tailwindcss-animate`.
- **UI primitives:** shadcn/ui — components are copied into the repo, not imported as a package.
- **XLSX writer:** `xlsx` (SheetJS) pinned to `^0.18.5` from the npm registry.
- **Cache:** Vercel KV.
- **Analytics:** Plausible (env-gated via `NEXT_PUBLIC_PLAUSIBLE_DOMAIN`).
- **Config:** `next.config.ts` (TS) kept minimal — only `reactStrictMode: true` until a concrete need surfaces.

**Consequences.**
- *Positive.* Vercel + Next 15 gives edge-runtime route handlers, ISR for the SEO variants, and zero-cost hobby hosting through MVP. Tailwind v3.4 + shadcn/ui matches the ecosystem most snippets/AI suggestions still target, so iteration is fast. SheetJS 0.18.5 from the registry keeps the lockfile clean and avoids the SheetJS CDN tarball footgun for now.
- *Negative.* Tailwind v4 and the SheetJS CDN tarball are both better long-term defaults; we'll pay a small migration cost when we adopt them. Vercel KV is a soft lock-in — if we leave Vercel, the cache layer needs a one-leaf swap (acceptable; the surface is `get/set` keyed by the slug from D-017).
- *Reversibility.* Framework swap = expensive (rewrite). Tailwind v3 → v4 = mechanical (codemod). KV provider swap = one leaf at the cache interface. SheetJS registry → CDN = one-line import unchanged.

### ADR-002 — Mock-first SemanticForce contract: our types are the contract, not theirs

**Status:** Accepted (2026-05-10). Supersedes nothing. Folds in D-002, D-003, D-014, D-015, D-016. Phase 4 (L4.1) is the human-gated point where this ADR's "real-creds" branch gets exercised.

**Context.** SemanticForce is the only data source for v1 — no direct scraping (PLAN.md, anti-audience). We do not yet have SF credentials; real creds arrive in a human-gated Phase 4 leaf. The autonomous routine and any future contributor must be able to build, test, and ship the entire MVP — UI, exporters, cache, rate-limiter, SEO pages — against deterministic data, then flip a single environment variable when creds land. SF's wire schema is unknown to us, will probably change once or twice between the public docs and reality, and is not something callers (route handlers, exporters, UI) should be coupled to.

**Decision.**
- The contract callers depend on is **ours**: `Review`, `PlaceMeta`, `GetReviewsResponse` in `lib/semanticforce/types.ts`, and the single client method `getReviews({placeId, limit?, after?}) → Promise<GetReviewsResponse>` in `lib/semanticforce/client.ts`.
- The client reads `SF_API_KEY` and `SF_API_BASE` from the environment. **`SF_API_KEY` unset ⇒ fixture mode**: the client serves from `mocks/semanticforce/{small,mid,large}-business.json`. This is the default state of the repo.
- Fixtures are produced by `scripts/gen-fixtures.py` — a deterministic seeded generator (not hand-written) for mid (80) and large (500); `small-business.json` stays hand-curated. The generator lives outside the Next.js source tree so it never ships to Vercel. Distribution targets (13 langs, ~40% en, 60/20/10/5/5 rating skew, ~15% photos, owner-response chance scaling with negativity) are encoded in one place.
- Errors surface as a single `SemanticForceError` class with a coarse `code` enum: `rate_limited` | `not_found` | `unauthorized` | `bad_request` | `upstream_error` | `unknown`. Network failures and 5xx both map to `upstream_error` so retry policy is one branch.
- Pagination: `limit` clamped to `[1, 100]` (default 50) inside the client. Fixture cursor is `btoa(JSON.stringify({offset:N}))` (edge-safe — no `Buffer`); the real-API cursor is opaque pass-through.
- When real creds land (L4.1), only `client.ts` changes; `types.ts`, callers, and exporters stay put. Any schema delta is documented in `docs/semanticforce-api.md` rather than leaking into call sites.

**Consequences.**
- *Positive.* Every Phase 1–3 leaf is unblocked without credentials. Diffs against fixtures are byte-stable so reviews focus on intentional changes. The coarse error enum stays valid no matter what shape SF's real error envelope takes. Edge-safe cursor encoding means no surprises moving to edge runtime.
- *Negative.* When creds arrive, our schema guess will be wrong somewhere — that adapter work is real. The "fixtures match reality" assumption is a known unknown until L4.1. Fixture mode also masks rate-limit and pagination edge cases that only show up against the live API (mitigated by `HARD_CAP_REVIEWS = 5_000`, `HARD_CAP_PAGES = 50` from D-019).
- *Reversibility.* Free — flip `SF_API_KEY` and the client switches branches. Fixtures stay in the repo as offline-dev / CI ammo even after creds land.

### ADR-003 — Export-format defaults: CSV (UTF-8 BOM + CRLF + QUOTE_ALL), JSON, XLSX

**Status:** Accepted (2026-05-10). Supersedes nothing. Folds in D-005. Implementation lands in L2.6 (CSV) and L2.7 (XLSX); JSON is `Response.json()`.

**Context.** The audience is local-SEO consultants and SMB owners pulling reviews for client work, sentiment analysis, marketing quotes, and backup. The dominant downstream tool is Excel (often on Windows, often via "double-click the CSV"). Reviews contain unicode the Excel CSV path historically mishandles: emoji, smart quotes, em dashes, accented Latin, CJK, RTL scripts. We can either pick defaults that just work in Excel, or push every user to learn the import wizard. The same fixture data is intentionally seeded with awkward unicode (D-014) so the writer is stress-tested on every change.

**Decision.**
- **CSV** (`lib/export/csv.ts`, L2.6): UTF-8 with **BOM**, **CRLF** line endings, **QUOTE_ALL** every field. One row per review; header row first. Per project memory `feedback_csv_ascii_for_excel`, this is the combination that makes Excel-on-Windows render unicode correctly without prompting the import wizard.
- **JSON**: `application/json`, the same `GetReviewsResponse` shape the client returns — `{reviews: Review[], place: PlaceMeta, next_cursor?: string}`. No transformation; no flattening. JSON consumers want the structure, not a denormalised export.
- **XLSX** (`lib/export/xlsx.ts`, L2.7): SheetJS `xlsx` writes a single worksheet, one row per review, frozen header (`ySplit: 1`), sensible per-column widths. Unicode is native to XLSX so no BOM/encoding gymnastics.
- **Filename convention** for downloads: `google-reviews-<slug>-<YYYYMMDD>.<ext>`, where `<slug>` is the lowercased URL-safe place-id slug from D-017. Stable across sessions so users can diff exports.
- **Format selection** is a query param `format=csv|json|xlsx` on `app/api/reviews/route.ts` (L2.2). CSV is the default — the audience and the head-term query both point at CSV.
- Errors during export do **not** produce a partial file. The route returns the `SemanticForceError`-shaped envelope from ADR-002 with the appropriate HTTP status; the UI surfaces it without writing a file.

**Consequences.**
- *Positive.* CSV double-clicks correctly in Excel-on-Windows even with multi-language review bodies. XLSX gives users a "looks like a spreadsheet" path without LibreOffice quirks. JSON is structurally identical to what the client/cache emits, so analysts can pipe it into anything. The seeded-unicode fixtures from D-014 mean every CSV writer change must survive Excel-realistic input.
- *Negative.* QUOTE_ALL bloats CSV file size ~10–20%. CRLF makes the file mildly annoying on Unix tools that don't auto-handle it (rare; most do). Locking the export shape early means a future "flat denormalised CSV with author / rating / text columns only" request becomes a new format flag rather than a default change.
- *Reversibility.* CSV defaults are one constant in `lib/export/csv.ts`; XLSX styling is one config object. Adding a fourth format = one writer module + one switch arm. No callers depend on internal export details, only on the `format` query param.

## 2026-05-13

- **D-035** `robots.ts` + `sitemap.ts` use Next 15 file conventions in `app/` (not static `public/` files). Rationale: Next regenerates both at build time from a typed function — once L3.1 variant pages exist, the sitemap walker can read the SEO-variants list and emit entries without a separate file update. Site URL reads from `NEXT_PUBLIC_SITE_URL` with a `https://googlereviewsdownload.com` fallback so the build is green pre-domain-purchase (L5.1); the fallback is harmless because robots/sitemap are never crawled until DNS resolves. `/api/` is the only Disallow — the page routes are crawlable.
- **D-036** L3.2 split into L3.2a (robots + sitemap, root-only — landed) and L3.2b (JSON-LD `FAQPage` schema). L3.2b is deferred because it needs both L3.1 variant pages (blocked on L1.6b corgi pass) and L3.3 FAQ content. Schema-without-content is worthless to crawlers, so we wait.
- **D-037** L4.2 healthcheck reports `mode` (`fixture`/`live`) inferred from `SF_API_KEY` presence rather than asking the client which class it instantiated — the `SemanticForceClient` interface intentionally hides Fixture-vs-Http, and the env var is the same signal `createSemanticForceClient()` itself branches on, so the two cannot disagree.
